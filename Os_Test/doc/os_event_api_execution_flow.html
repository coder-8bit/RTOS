GIÁO TRÌNH CHUYÊN SÂU EVENT API TRONG MINI RTOS

Từ event bit trong TCB đến task block/unblock qua scheduler

Tài liệu này bám sát đúng code trong repository Os_Test sau khi thêm 4 API nền: WaitEvent(), SetEvent(), ClearEvent(), GetEvent(). Đây là phần học nền RTOS, chưa đi vào đầy đủ StatusType hoặc error semantics của OSEK.

Một câu chốt bản chất của Event: Event là các bit tín hiệu gắn trực tiếp với từng task. Task có thể ngủ bằng WaitEvent(mask); task khác hoặc ISR có thể báo bằng SetEvent(task, mask); khi bit khớp mask đang chờ, task được đưa từ WAITING về READY.

1. Mục tiêu và bức tranh tổng thể

Sau bài này, người học cần hiểu được vì sao event là bước tiếp theo tự nhiên sau scheduler. Trước đó task chỉ chạy theo autostart, alarm hoặc schedule table. Với event, task có thể chủ động ngủ khi chưa có việc, và chỉ quay lại ready queue khi có tín hiệu phù hợp.

Chuỗi runtime cần nắm:
Task Consumer chạy → WaitEvent(mask) → chưa có event → state = WAITING → scheduler chạy Producer → Producer SetEvent(Consumer, mask) → Consumer state = READY → scheduler preempt Producer nếu Consumer ưu tiên cao hơn → Consumer GetEvent/ClearEvent → Consumer xử lý xong
APIVai tròCó block?Có thể gây schedule?

SetEvent(tid, mask)Báo event cho task đích.Không.Có, nếu task đích đang chờ và được unblock.
ClearEvent(mask)Xóa các bit event đã xử lý của task hiện tại.Không.Không.
GetEvent(tid, *mask)Đọc event_set hiện tại của task đích.Không.Không.

1.1 Năng lực đầu ra sau bài học

Bài này không chỉ yêu cầu học viên nhớ tên 4 API. Sau khi học xong, học viên phải đọc được code kernel và tự giải thích chính xác CPU đang chạy ở đâu, task nào còn nằm trong ready queue, task nào đang ngủ, event bit nào đang pending, và vì sao PendSV có hoặc không xảy ra.

1.2 Kiến thức cần có trước khi học

  • Đã hiểu task state cơ bản: DORMANT, READY, RUNNING, WAITING.
  • Đã hiểu ready queue theo priority và FIFO trong cùng priority.
  • Đã hiểu PendSV là exception thực hiện context switch, còn C scheduler chỉ chọn g_next.
  • Đã hiểu critical section tối thiểu bằng PRIMASK.
  • Đã biết đọc log Renode từ USART1.

1.3 Câu hỏi trung tâm của bài

Nếu task đang chạy không có việc để làm, nó nên bận chờ bằng vòng lặp polling, hay nên tự chuyển sang WAITING để CPU chạy task khác?

Event API trả lời câu hỏi này. Nó biến một task từ kiểu "luôn chạy để kiểm tra cờ" thành kiểu "ngủ đến khi có tín hiệu". Đây là một bước rất quan trọng trong RTOS vì nó giảm lãng phí CPU, giảm jitter cho task khác, và làm flow phần mềm rõ ràng hơn.

2. Bản chất của event bitmask

2.1 Event không phải queue

Event trong bài này là một bitmask. Mỗi bit đại diện cho một điều kiện đã xảy ra. Nếu cùng một bit được set nhiều lần trước khi task clear, trạng thái vẫn chỉ là 1. Vì vậy event phù hợp cho tín hiệu trạng thái như "RX done", "timeout", "button pressed"; không phù hợp nếu cần đếm số lần xảy ra.

enum {
    EVENT_RX_DONE = (1u << 0),
    EVENT_TIMEOUT = (1u << 1)
};
Giá trịÝ nghĩaDiễn giải
0x00000000Không có event pendingTask gọi WaitEvent() sẽ block nếu mask cần chờ chưa khớp.
0x00000001EVENT_RX_DONEConsumer có dữ liệu RX cần xử lý.
0x00000002EVENT_TIMEOUTConsumer có timeout cần xử lý.
0x00000003Cả hai bit cùng pendingTask có thể xử lý từng bit rồi clear từng phần.
Event không tự mất. SetEvent() chỉ bật bit. Sau khi xử lý, task phải gọi ClearEvent(). Nếu không clear, lần WaitEvent() tiếp theo sẽ return ngay vì bit vẫn còn pending.

2.3 Auto-clear hay manual-clear?

Bản hiện tại dùng manual-clear: WaitEvent() không xóa bit, task phải tự gọi ClearEvent() sau khi xử lý. Đây là cách giúp task kiểm tra được nhiều bit cùng lúc và quyết định clear bit nào.

KiểuCách hoạt độngƯu điểmRủi ro
Manual-clearSetEvent bật bit, task xử lý xong gọi ClearEvent.Rõ ràng, task kiểm soát thời điểm xóa.Quên clear thì WaitEvent sau return ngay.
Auto-clearKernel tự xóa bit khi WaitEvent return.Code task ngắn hơn.Dễ mất thông tin nếu task cần đọc nhiều bit hoặc xử lý theo thứ tự riêng.

2.4 Công thức bitmask cần nhớ

/* Set một hoặc nhiều bit */
event_set |= mask;

/* Kiểm tra có ít nhất một bit khớp */
if ((event_set & wait_mask) != 0u) { ... }

/* Clear một hoặc nhiều bit */
event_set &= ~mask;

/* Chờ một trong hai event */
WaitEvent(EVENT_RX_DONE | EVENT_TIMEOUT);

3.3 Data ownership: ai sở hữu event?

Trong thiết kế này, event không phải object độc lập. Event thuộc về task nhận. Vì vậy Task_EventConsumer sở hữu event_set của nó, còn Task_EventProducer chỉ có quyền bật bit thông qua SetEvent(TASK_EVENT_CONSUMER, mask).

Task State Machine với Event Blocking

4. Phân tích 4 API

4.1 WaitEvent(mask)

WaitEvent() chỉ tác động lên task hiện tại. Nếu event đã có sẵn, hàm return ngay. Nếu chưa có, task bị chuyển sang OS_WAITING và scheduler chọn task khác.

void WaitEvent(EventMaskType mask)
{
    if ((current->event_set & mask) != 0u) {
        return;
    }

    current->wait_mask = mask;
    current->state = OS_WAITING;
    os_schedule();
}
Điểm quan trọng: task đang WAITING không nằm trong ready queue. Nó chỉ quay lại ready queue khi SetEvent() phát một bit khớp với wait_mask.

4.2 SetEvent(tid, mask)

SetEvent() OR bit vào task đích. Nếu task đích đang chờ và có bit khớp, task được chuyển về READY, enqueue theo priority, rồi scheduler có thể dispatch.

task->event_set |= mask;

if ((task->state == OS_WAITING) &&
    ((task->event_set & task->wait_mask) != 0u)) {
    task->wait_mask = 0u;
    task->state = OS_READY;
    os_ready_add(task->id, task->current_priority);
    os_schedule();
}

4.3 GetEvent(tid, *mask)

GetEvent() chỉ copy event_set ra biến của caller. Nó không clear, không block, không gọi scheduler.

EventMaskType events;
GetEvent(TASK_EVENT_CONSUMER, &events);

4.4 ClearEvent(mask)

ClearEvent() xóa bit event của task hiện tại bằng phép AND với bit đảo.

current->event_set &= ~mask;
Trong bản nền này, API đang dùng kiểu void để đồng bộ với style kernel hiện tại. Các rule lỗi kiểu gọi từ ISR, task ID sai, access right, extended/basic task sẽ để sang cụm OSEK API sau.

5.2 Vì sao Consumer ưu tiên cao hơn Producer?

Đây là cách demo rõ nhất cho unblock + preemption. Consumer chạy trước, gọi WaitEvent() rồi block. Producer bắt đầu chạy và gọi SetEvent(). Vì Consumer có priority cao hơn Producer, scheduler có quyền ngắt Producer để Consumer xử lý event ngay.

6.4 Điều gì xảy ra nếu event đã pending trước WaitEvent?

Trường hợp này rất quan trọng khi dạy học viên vì nó chứng minh event là sticky bit. Nếu Producer gọi SetEvent() trước khi Consumer gọi WaitEvent(), event_set đã có bit. Khi Consumer gọi WaitEvent(EVENT_RX_DONE), điều kiện (event_set & mask) != 0 đúng, nên Consumer không block.

/* Flow khác nhưng vẫn đúng */
SetEvent(TASK_EVENT_CONSUMER, EVENT_RX_DONE);

/* Sau đó Consumer chạy */
WaitEvent(EVENT_RX_DONE);  /* return ngay */
GetEvent(TASK_EVENT_CONSUMER, &events); /* events có bit RX */
Đây là cơ chế tránh "lost wakeup" ở mức semantic. Event đến trước vẫn được lưu lại trong event_set cho đến khi task nhận clear.

7. Race safety và critical section

Cả bốn API đều chạm vào state dùng chung của kernel: event_set, wait_mask, state, ready queue. Vì các field này có thể bị truy cập từ task context và sau này từ ISR, mỗi API đều dùng critical section nội bộ.

uint32_t primask = os_irq_save();
/* đọc/sửa TCB và ready queue */
os_irq_restore(primask);

10. Ứng dụng thực tế trong hệ automotive

10.1 UART RX: ISR đánh thức parser task

Ứng dụng kinh điển nhất là UART RX. ISR không nên parse frame dài. ISR chỉ ghi byte vào ring buffer và set event để task xử lý sau.

Use Case: UART RX ISR → Parser Task
sequenceDiagram participant UART as USART1_IRQHandler participant RB as Ring Buffer participant OS as SetEvent() participant Parser as Task_UartParser UART->>RB: push received byte UART->>OS: SetEvent(Task_UartParser, EV_UART_RX) OS->>Parser: unblock when waiting Parser->>Parser: WaitEvent(EV_UART_RX) returns Parser->>RB: pop bytes and parse frame Parser->>OS: ClearEvent(EV_UART_RX)

Thiết kế đúng là event chỉ báo "có dữ liệu", còn dữ liệu nằm trong ring buffer. Nếu ISR nhận 10 byte liên tiếp, event bit vẫn chỉ là 1, nhưng ring buffer giữ đủ 10 byte.

10.2 CAN RX: CanIf báo event cho application task

Trong gateway automotive, CAN driver hoặc CanIf nhận PDU rồi đánh thức task xử lý application. Event giúp tách phần thời gian gấp trong ISR khỏi phần xử lý nghiệp vụ.

10.3 Timeout và watchdog supervision

Alarm hoặc ScheduleTable có thể gọi SetEvent(Task_X, EV_TIMEOUT). Task chờ cả data và timeout trong một API:

WaitEvent(EV_DATA_READY | EV_TIMEOUT);
GetEvent(TASK_X, &events);

if ((events & EV_DATA_READY) != 0u) {
    ClearEvent(EV_DATA_READY);
    ProcessData();
}

if ((events & EV_TIMEOUT) != 0u) {
    ClearEvent(EV_TIMEOUT);
    HandleTimeout();
}

Đây là pattern rất quan trọng: một task có thể chờ nhiều nguyên nhân đánh thức, rồi phân nhánh xử lý theo bit.

14. Điểm dễ nhầm và kết luận

  • SetEvent() không trực tiếp gọi function của task đích. Nó chỉ set bit và có thể đưa task về ready queue.
  • WaitEvent() không clear event. Nếu event đã pending, hàm return ngay.
  • GetEvent() chỉ đọc, không thay đổi state.
  • ClearEvent() nên được gọi sau khi task đã xử lý bit tương ứng.
  • Event bitmask không đếm số lần xảy ra. Set cùng một bit nhiều lần vẫn chỉ là bit 1.
  • Task đang WAITING không được scheduler chọn cho đến khi có event khớp.
Kết luận kỹ thuật: Sau bài này, kernel không còn chỉ là scheduler chạy task theo ready queue. Nó đã có primitive blocking đầu tiên: task có thể tự rời CPU khi chưa có việc và quay lại khi được báo event. Đây là nền quan trọng trước khi học OSEK event service đầy đủ.
