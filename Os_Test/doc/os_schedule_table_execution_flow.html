Từ bảng cấu hình tĩnh đến ActivateTask() tại Expiry Point — xuyên suốt SysTick ISR

Tài liệu này bám sát đúng code trong repository Os_Test. Mục tiêu không chỉ là mô tả "ScheduleTable là gì", mà là chỉ rõ dữ liệu cấu hình nằm ở đâu, ai đọc bảng nào, tick được đếm ra sao, và cuối cùng task nào được kích hoạt tại thời điểm nào.

Một câu chốt bản chất của ScheduleTable: Application không tự gọi ActivateTask() theo lịch trình. Thay vào đó, ScheduleTable module đếm tick từ SysTick ISR, và khi offset trong chu kỳ trùng với một expiry point đã cấu hình sẵn, nó tự động gọi ActivateTask(). Scheduler core vẫn là nơi quyết định task nào thật sự chạy.

2. ScheduleTable là gì ở góc nhìn OSEK/AUTOSAR

2.1 Bản chất của ScheduleTable

ScheduleTable là cơ chế time-triggered scheduling trong OSEK/AUTOSAR OS. Nó cho phép hệ thống kích hoạt một chuỗi các task theo lịch trình cố định, lặp lại theo chu kỳ, mà không cần application code chủ động quản lý thời gian.

Đặc tínhMô tảÝ nghĩa trong demo
Kiểu lập lịchTime-triggered (dựa trên thời gian)Các task được kích hoạt tại offset cố định trong chu kỳ.
Đơn vị thời gianTick (OS counter)1 tick = 1 ms (OS_TICK_HZ = 1000).
RepeatingCó thể lặp vô hạn hoặc one-shotDemo đặt repeating = 1 → lặp mỗi 40 ms.
Expiry PointsCác mốc thời gian trong chu kỳ cần thực thi actionOffset 0 → Low task, Offset 10 → High task.
Action hỗ trợActivateTask (bản hiện tại)Mỗi expiry point gọi ActivateTask(task_id).

2.2 Tại sao cần ScheduleTable khi đã có Alarm?

Tiêu chíAlarmScheduleTable
Số actionMột alarm → một target taskMột table → nhiều expiry points → nhiều tasks
Đồng bộ offsetMỗi alarm đếm độc lập, offset dễ trôiTất cả expiry points đồng bộ trong cùng một chu kỳ
Cấu hìnhĐơn giản: delay + cycle + targetPhức tạp hơn: duration + N expiry points + repeating
Use casePeriodic task đơn, watchdog, timeoutChuỗi task phải chạy theo trình tự thời gian cố định
AUTOSAROSEK legacy, vẫn hỗ trợAUTOSAR OS mở rộng, khuyến nghị cho time-triggered
Điểm cốt lõi: Alarm phù hợp cho tác vụ đơn lẻ (1 alarm = 1 task). Nhưng khi cần một chuỗi nhiều task chạy theo thứ tự thời gian đồng bộ trong cùng chu kỳ (ví dụ: đọc sensor → xử lý → gửi CAN), ScheduleTable là giải pháp chuẩn vì tất cả offset đều tính từ cùng một gốc thời gian, không bị trôi riêng lẻ.

2.3 Các trạng thái runtime của ScheduleTable

RUNNING (delay phase)Table đang đếm ngược start_delay_ticks→ RUNNING (cycle phase) khi countdown = 0
RUNNING (cycle phase)Table đếm elapsed_ticks, fire expiry points→ Reset offset 0 (nếu repeating) hoặc → STOPPED (one-shot)

3. Kiến trúc phần mềm và vị trí ScheduleTable trong OS

3.1 Ba tầng chính của OS

ScheduleTable nằm giữa tầng cấu hình tĩnh và tầng kernel core. Nó không tự dispatch task, mà chỉ bơm activation vào ready queue thông qua API ActivateTask().

3.2 Nguyên tắc phân tách trách nhiệm

ScheduleTable module chỉ làm duy nhất một việc: Đếm tick, kiểm tra expiry point, và gọi ActivateTask(). Nó KHÔNG quyết định task nào chạy, KHÔNG trigger PendSV, và KHÔNG thao tác ready queue trực tiếp.

Scheduler core (trong os_kernel.c) là nơi duy nhất quyết định dispatch dựa trên ready queue, priority, và policy preemption (Full/Non/Mixed).

5. Luồng thực thi đầy đủ từ OS_Init đến Task chạy

5.1 Giai đoạn 1: Khởi tạo — OS_Init()

// os_kernel.c — OS_Init() (dòng 684-764)
void OS_Init(void)
{
    // 1. Reset trạng thái toàn bộ kernel
    g_current = NULL;
    g_next = NULL;
    g_tick_count = 0u;

    // 2. Cấu hình port Cortex-M3 (SysTick, priority exception)
    os_port_init();

    // 3. Reset ready queue, alarm, ScheduleTable
    os_ready_queue_reset();
    os_alarm_reset_all();
    os_sched_table_reset_all();         // ← Reset runtime state ScheduleTable

    // 4. Dựng TCB cho toàn bộ task từ bảng config
    for (tid = 0; tid < OS_MAX_TASKS; ++tid)
        os_task_setup_from_cfg(tid);

    // 5. Nạp task autostart vào ready queue
    // Task_Init (CRITICAL) và Task_Idle được đưa vào queue

    // 6. Chọn task đầu tiên (Task_Init) cho bootstrap
    g_current = &tcb[first_id];

    // 7. Nạp các nguồn activation tĩnh
    os_alarm_autostart_all();
    os_sched_table_autostart_all();     // ← Autostart ScheduleTable!
}

5.2 Giai đoạn 2: Autostart ScheduleTable

// os_sched_table.c — autostart (dòng 200-210)

5.3 Giai đoạn 3: Start runtime của ScheduleTable

// os_sched_table.c — StartScheduleTableRel() (dòng 130-156)
void StartScheduleTableRel(uint8_t sid, uint32_t start_delay_ms)
{
    // Validate: expiry points != NULL, duration > 0
    OsScheduleTable_t *rt = &g_sched_table_rt[sid];

    rt->active = 1u;                    // Đánh dấu đang chạy
    rt->repeating = cfg->repeating;     // Copy repeating từ config
    rt->current_expiry_idx = 0u;        // Con trỏ bắt đầu từ expiry 0
    rt->elapsed_ticks = 0u;             // Offset = 0
    rt->duration_ticks = ms_to_ticks(cfg->duration_ms);   // = 40
    rt->start_delay_ticks = ms_to_ticks(start_delay_ms);  // = 220
}

5.5 Giai đoạn 5: SysTick ISR xử lý tick

// os_kernel.c — os_on_tick() (dòng 631-674)
void os_on_tick(void)
{
    os_isr_enter();                          // Tăng interrupt nesting
    g_tick_count++;                          // Đếm tick toàn cục

    // Bước 1: Xử lý alarm
    for (i = 0; i < OS_MAX_ALARMS; ++i) { /* ... */ }

    // Bước 2: Xử lý ScheduleTable
    os_sched_table_on_tick();                // ← ScheduleTable tick handler!

    os_isr_exit();                           // Giảm nesting, gọi os_schedule()
}
Thứ tự xử lý trong os_on_tick(): Alarm được xử lý trước, ScheduleTable sau. Cả hai đều gọi ActivateTask() nhưng scheduler chỉ chạy một lần duy nhất ở cuối ISR (os_isr_exit()) khi interrupt nesting giảm về 0.

6. Phân tích chi tiết xử lý tick: từ SysTick ISR đến ActivateTask

6.1 os_sched_table_on_tick() — Tim đập của ScheduleTable

// os_sched_table.c — tick handler (dòng 216-255)
void os_sched_table_on_tick(void)
{
    for (sid = 0; sid < OS_MAX_SCHEDULE_TABLES; ++sid) {
        OsScheduleTable_t *rt = &g_sched_table_rt[sid];

        if (rt->active == 0u) continue;          // Bỏ qua table không active

        // ========= GIAI ĐOẠN 1: Countdown start delay =========
        if (rt->start_delay_ticks > 0u) {
            rt->start_delay_ticks--;              // Giảm 1 mỗi tick
            if (rt->start_delay_ticks == 0u) {
                // Delay xong! Reset offset và fire expiry points tại offset 0
                rt->elapsed_ticks = 0u;
                rt->current_expiry_idx = 0u;
                os_sched_table_fire_due_points(sid);
                os_sched_table_finish_cycle(sid);
            }
            continue;                             // Chưa vào chu kỳ, skip
        }

        // ========= GIAI ĐOẠN 2: Đếm offset trong chu kỳ =========
        rt->elapsed_ticks++;                      // Tăng offset
        os_sched_table_fire_due_points(sid);      // Kiểm tra expiry points
        os_sched_table_finish_cycle(sid);          // Kiểm tra hết chu kỳ chưa
    }
}

6.2 os_sched_table_fire_due_points() — Gọi ActivateTask tại expiry point

// os_sched_table.c — fire due points (dòng 74-92)
static void os_sched_table_fire_due_points(uint8_t sid)
{
    const OsScheduleTableConfig_t *cfg = &g_os_schedule_table_cfg[sid];
    OsScheduleTable_t *rt = &g_sched_table_rt[sid];

    while (rt->current_expiry_idx < cfg->num_expiry_points) {
        const OsScheduleTableExpiryPointCfg_t *point =
            &cfg->expiry_points[rt->current_expiry_idx];

        if (point->offset_ticks > rt->elapsed_ticks)
            break;                    // Chưa đến offset này, dừng

        ActivateTask(point->task_id); // ← KÍCH HOẠT TASK!
        rt->current_expiry_idx++;     // Chuyển sang expiry point tiếp
    }
}
Tại sao dùng <= thay vì ==?
Dùng so sánh offset_ticks > elapsed_ticks (tức fire khi <=) để nếu một tick bị trễ hoặc xử lý chậm, expiry point vẫn không bị bỏ sót. Đây là kỹ thuật phòng thủ phổ biến trong embedded RTOS.

6.3 os_sched_table_finish_cycle() — Kết thúc chu kỳ

// os_sched_table.c — finish cycle (dòng 100-122)
static void os_sched_table_finish_cycle(uint8_t sid)
{
    OsScheduleTable_t *rt = &g_sched_table_rt[sid];

    if (rt->elapsed_ticks < rt->duration_ticks)
        return;                             // Chưa hết chu kỳ

    if (rt->repeating != 0u) {
        // REPEATING: Reset offset về 0, fire ngay expiry points offset 0
        rt->elapsed_ticks = 0u;
        rt->current_expiry_idx = 0u;
        os_sched_table_fire_due_points(sid);
    } else {
        // ONE-SHOT: Dừng hẳn table
        rt->active = 0u;
        rt->current_expiry_idx = 0u;
        rt->elapsed_ticks = 0u;
        rt->start_delay_ticks = 0u;
    }
}

6.4 ActivateTask() — Cầu nối giữa ScheduleTable và Scheduler

// os_kernel.c — ActivateTask() (dòng 508-555)
void ActivateTask(uint8_t tid)
{
    TCB_t *task = &tcb[tid];

    if (task->activation_count >= task->max_activations) return;
    task->activation_count++;

    if (task->state == OS_DORMANT) {
        task->context_needs_init = 1u;    // Dựng lại stack frame
        task->state = OS_READY;
        os_ready_enqueue_tail(task->current_priority, task->id);
        // → Task nằm trong ready queue, chờ scheduler pick
    }

    // Nếu không trong ISR → gọi os_schedule() ngay
    // Nếu đang trong ISR → os_isr_exit() sẽ gọi os_schedule() sau
}
Luồng đi hoàn chỉnh trong 1 tick:
SysTick_Handler() → os_on_tick() → os_sched_table_on_tick() → fire_due_points() → ActivateTask(task_id) → enqueue vào ready queue → os_isr_exit() → os_schedule() → os_should_preempt() → os_dispatch() → PendSV context switch → Task chạy

11. Các điểm dễ nhầm và kết luận kỹ thuật

11.1 Sáu điểm dễ nhầm nhất

  • ScheduleTable KHÔNG tự dispatch task. Nó chỉ gọi ActivateTask() để đưa task vào ready queue. Scheduler core là nơi duy nhất quyết định context switch, dựa trên priority và policy (Full/Non/Mixed Preemptive).
  • start_delay ≠ offset. start_delay_ms là thời gian chờ trước khi bắt đầu chu kỳ đầu tiên. offset_ticks trong expiry point là vị trí trong chu kỳ. Hai khái niệm hoàn toàn khác nhau.
  • start_delay = 0 không fire ngay. Code quy đổi delay 0 ms thành 1 tick (ms_to_ticks(1)) để tránh gọi ActivateTask() ngay trong OS_Init(). Tick đầu tiên mới fire.
  • Expiry point offset 0 fire ngay khi countdown = 0. Không cần chờ thêm 1 tick nữa. Logic fire chạy ngay trong cùng tick mà start_delay_ticks chạm 0.
  • Repeating fire offset 0 ngay tại boundary. Khi elapsed_ticks == duration_ticks, finish_cycle() reset elapsed về 0 và gọi fire_due_points() luôn. Expiry point tại offset 0 của chu kỳ mới fire ngay trong cùng tick kết thúc chu kỳ cũ.
  • Alarm và ScheduleTable cùng chạy trong os_on_tick() nhưng KHÔNG xung đột. Alarm xử lý trước, ScheduleTable sau. Tất cả các ActivateTask() đều enqueue vào ready queue. Scheduler chỉ chạy một lần ở cuối ISR, chọn task priority cao nhất từ queue.

11.2 Kết luận cuối cùng

Điều quan trọng nhất để hiểu ScheduleTable trong OSEK/AUTOSAR OS là: ScheduleTable là một "bộ hẹn giờ thông minh" với nhiều mốc thời gian, không phải là một scheduler riêng biệt. Cụ thể:

  • Cấu hình tĩnh (os_app_cfg.c) định nghĩa table với các expiry points chứa offset + task_id.
  • OS_Init() đọc bảng config, tạo runtime state, và autostart table nếu cần.
  • SysTick ISR gọi os_sched_table_on_tick() mỗi 1 ms để giảm countdown hoặc tăng offset.
  • ScheduleTable module kiểm tra expiry point và gọi ActivateTask() khi đến offset.
  • Scheduler core chọn task có priority cao nhất từ ready queue để dispatch.
  • PendSV thực hiện context switch thật sự ở mức assembly.

Nắm được chuỗi này thì việc mở rộng ScheduleTable với các tính năng AUTOSAR nâng cao như SetEvent(), synchronization, next table chaining, hay implicit/explicit sync sẽ trở nên tự nhiên hơn rất nhiều.
